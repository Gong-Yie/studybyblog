2.1 进程描述

前趋图

前趋图是一个有向无循环图，用于描述程序各部分之间的执行先后关系。
结点表示语句、程序段或进程；有向边表示前趋关系。

程序的顺序执行

通常一个程序可分成若干个程序段，它们必须按照某种先后次序执行。仅当前一个操作执行完毕，后一个操作才能开始。

顺序执行的特征：

  • 顺序性：各程序段严格按照先后次序执行
  • 封闭性：程序独占全机资源，执行结果不受外界影响
  • 可再现性：只要初始条件相同，多次执行结果相同

程序的并发执行

程序的并发执行：一组在逻辑上互相独立的程序或程序段，在执行过程中，其执行时间在客观上互相重叠。

并发执行的特征：

  • 间断性/异步性：程序“走走停停”，失去原来的严格时序关系
  • 失去封闭性：多个程序共享系统资源，彼此可能影响
  • 不可再现性：并发执行顺序不同，结果可能不同
理解点：多道程序环境下，程序的执行属于并发执行。为了描述、控制这种动态执行过程，必须引入“进程”。

进程的引入

引入进程的目的：为了使多道程序能够并发执行，提高资源利用率和系统效率。

进程的定义：进程是程序的一次执行过程，是系统进行资源分配和调度的基本单位。

进程的结构特征：由程序段、数据段、进程控制块PCB三部分构成，又称进程映像。

进程和程序

对比项 程序 进程
性质 静态 动态
存在形式 软件资料，可长期保存 一次执行过程，有生命周期
组成 指令集合 程序段、数据段、PCB
对应关系 一个程序可对应多个进程 一个进程可执行一个或多个程序

2.2 进程状态与PCB

进程的三种基本状态

一个进程从创建、运行到消亡，可用一组状态描述。最基本的三种状态：

  • 就绪态：已经具备运行条件，只差CPU
  • 运行态：正在CPU上执行
  • 阻塞态：等待某事件发生，暂时不能运行

状态转换条件：

转换 原因
就绪 –> 运行 被进程调度程序选中，获得CPU
运行 –> 就绪 时间片用完，或被更高优先级进程抢占
运行 –> 阻塞 请求I/O或等待事件
阻塞 –> 就绪 等待事件发生，或I/O完成

挂起状态

挂起状态/静止状态：程序运行期间，由于某种需要，将进程暂停执行，使其静止下来。

引入挂起状态的原因：

  • 终端用户需要：调试程序时希望暂停进程
  • 父进程请求：父进程需要检查或协调子进程
  • 负荷调节：系统负荷过重时，将部分进程换出内存
  • 操作系统需要：检查资源使用或系统状态

创建状态和终止状态

  • 创建状态：进程正在被创建，尚未进入就绪队列
  • 终止状态：进程完成或被撤销，等待系统回收资源

进程控制块PCB

PCB是操作系统为了描述和控制进程运行而为每个进程定义的数据结构。
它记录了操作系统用于描述进程情况及控制进程运行所需的全部信息。

PCB的主要内容：

类别 内容
进程标识符 内部标识符PID、外部标识符
处理机状态 通用寄存器、指令计数器、程序状态字、用户栈指针
进程调度信息 进程状态、优先级、调度队列指针、事件
进程控制信息 程序和数据地址、资源清单、同步通信机制、链接指针

PCB的组织方式：

  • 线性方式：所有PCB组织在一张线性表中
  • 链接方式：按状态组织为就绪队列、阻塞队列、空白队列等
  • 索引方式：按状态建立不同索引表

2.3 进程控制

进程控制的主要任务：对系统中所有进程从产生、存在到消亡的全过程实行有效管理和控制。
进程控制是进程管理中最基本的功能。

原语

原语（Primitive）：由若干条指令组成、在管态下执行、常驻内存、用于完成系统特定功能的一个过程。

原语特点：

  • 执行过程不可分割
  • 执行期间不允许被中断
  • 通常用于进程控制、同步互斥等核心操作

进程创建

引起进程创建的事件：

  • 用户登录
  • 作业调度
  • 提供服务
  • 应用请求

创建过程：

  1. 申请空白PCB
  2. 为新进程分配资源
  3. 初始化PCB
  4. 将新进程插入就绪队列

进程终止

引起进程终止的事件：

  • 正常结束
  • 异常结束
  • 外界干预

终止过程：

  1. 根据进程标识符找到PCB
  2. 若进程正在运行，立即终止其执行
  3. 终止其子孙进程
  4. 释放资源
  5. 从所在队列中移出PCB

进程阻塞与唤醒

进程阻塞：正在执行的进程等待某事件尚未出现时，将自己阻塞起来。
进程唤醒：等待事件发生后，将阻塞进程转为就绪态。

可能引起阻塞的事件：

  • 请求共享资源失败
  • 等待某种操作完成
  • 新数据尚未到达
  • 等待新任务

进程挂起与激活

挂起：把指定进程或处于阻塞状态的进程调到外存，使其暂停运行。
激活：把被挂起进程重新调入内存，并恢复为就绪或阻塞状态。

2.4 进程同步

为什么需要进程同步

如果不能采取有效措施，对多个进程的运行进行妥善管理，必然会因为并发执行的随机性而造成错误结果。

进程同步：对多个相关进程在执行次序上进行协调。
同步的任务：使系统中各进程之间能有效共享资源、相互合作，从而使程序执行具有可再现性。

进程之间的两种制约关系

制约关系 含义 关键词
直接相互制约关系 进程之间因合作而形成先后执行关系 同步
间接相互制约关系 多个进程竞争独占型资源，必须互斥执行 互斥

临界资源和临界区

临界资源：一次只允许一个进程使用的共享资源，如打印机、绘图机、变量、数据等。
临界区：进程中访问临界资源的那段代码。

临界区的通用结构：

  • 进入区：检查能否进入临界区
  • 临界区：访问临界资源
  • 退出区：释放临界资源
  • 剩余区：执行其他代码

同步机制应遵循的规则：

  • 空闲让进：临界区空闲时，应允许一个请求进入的进程立即进入
  • 忙则等待：已有进程进入临界区时，其他进程必须等待
  • 有限等待：等待进入临界区的进程应在有限时间内进入
  • 让权等待：不能进入临界区时，应释放CPU，避免忙等

同步机制

常见同步机制：

  • 硬件方法：Test-and-Set、Swap
  • 信号量机制：整型信号量、记录型信号量、AND型信号量、信号量集
  • 管程机制

Test-and-Set和Swap

Test-and-Set指令：原子地检查和修改一个字的内容。
Swap指令：原子地交换两个变量的值。

硬件方法优点：实现简单，适合低层互斥。
硬件方法缺点：可能导致忙等，难以表达复杂同步关系。

信号量机制

信号量机制由Dijkstra提出，也称P、V操作。
基本思想：两个或多个进程通过简单的信号进行合作，一个进程可以在某位置等待，直到收到另一个进程发来的信号。

整型信号量：用一个非负整数表示可用资源数。
wait(S)：申请资源，S减1。
signal(S)：释放资源，S加1。

记录型信号量：在整型变量之外增加等待队列，避免进程长期忙等。

信号量的含义：

  • S.value > 0：表示可用资源数量
  • S.value = 0：表示没有可用资源，也没有等待进程
  • S.value < 0：其绝对值表示等待该资源的进程数

AND型信号量与信号量集

AND型信号量：用于处理一个进程需要同时获得多个临界资源的情况。
核心思想：要么一次性获得全部资源，要么一个也不占有，避免“占有部分资源再等待其他资源”造成死锁。

信号量集：对多个资源、多个数量的申请进行统一处理，适合资源种类和数量都比较复杂的场景。

信号量的应用

利用信号量实现进程互斥：
为临界资源设置一个互斥信号量mutex，初值为1。进入临界区前执行P(mutex)，退出临界区后执行V(mutex)。

利用信号量描述前趋关系：
如果S1必须先于S2执行，可设置同步信号量a，初值为0。S1完成后V(a)，S2执行前P(a)。

管程机制

管程定义了一个数据结构和一组能被并发进程执行的操作。
管程中的变量只能被管程中的操作访问，任何时候只允许一个进程在管程中操作。

管程特点：

  • 封装共享变量
  • 统一管理同步与互斥
  • 降低信号量使用错误的风险

2.5 经典进程的同步问题

在多道程序环境下，进程同步问题十分重要。经典同步问题的意义不只是“会做题”，更重要的是练熟两件事：

  • 判断哪些地方需要同步：谁必须等谁
  • 判断哪些地方需要互斥：谁不能同时进入

代表问题：

  • 生产者-消费者问题
  • 哲学家进餐问题
  • 读者-写者问题

生产者-消费者问题

问题描述：一组生产者向一组消费者提供产品，生产者和消费者共享一个有界缓冲池。生产者把产品放入缓冲池，消费者从缓冲池取出产品。

同步关系：

  • 缓冲池满时，生产者必须等待
  • 缓冲池空时，消费者必须等待

互斥关系：

  • 多个生产者/消费者访问缓冲池时，必须互斥

常用信号量：

信号量 初值 含义
empty n 空缓冲区数量
full 0 满缓冲区数量
mutex 1 对缓冲池互斥访问

消费者过程：

重点：资源信号量的P操作通常放在互斥信号量P操作之前。否则可能出现“拿着锁等待资源”的情况，使其他进程也无法进入临界区。

生产者-消费者问题中应注意：

  1. 互斥信号量的P、V操作必须成对出现
  2. 资源信号量empty、full的P、V操作也必须成对出现，但分别位于不同进程中
  3. P操作顺序不能随意交换，尤其要避免先P(mutex)再等待资源
  4. V操作顺序通常影响不大，但应保持逻辑清楚

哲学家进餐问题

问题描述：五个哲学家围坐圆桌，每人左右各有一支筷子。哲学家交替思考和进餐，进餐时必须同时拿到左右两支筷子。

若每个哲学家都先拿左筷子，再拿右筷子，可能出现所有人都拿到一支筷子并等待另一支筷子的情况，从而发生死锁。

避免死锁的方法：

  • 至多允许4个哲学家同时进餐
  • 奇数号哲学家先取左筷子，偶数号哲学家先取右筷子
  • 仅当左右两支筷子都可用时，才允许哲学家拿筷子

读者-写者问题

问题描述：一个数据对象可被多个进程共享。读者只读数据，写者要修改数据。

约束关系：

  • 允许多个读者同时读
  • 只允许一个写者写
  • 写者写时，不允许任何读者读，也不允许其他写者写

读者优先算法常用变量：

变量/信号量 初值 含义
readcount 0 当前正在读的读者数量
rmutex 1 对readcount互斥访问
wmutex 1 写者与其他读写进程之间互斥

读者进程思路：

  1. 进入时互斥修改readcount
  2. 第一个读者负责P(wmutex)，阻止写者
  3. 退出时互斥修改readcount
  4. 最后一个读者负责V(wmutex)，允许写者

写者进程思路：

  1. P(wmutex)
  2. 写数据
  3. V(wmutex)

注意：读者优先可能导致写者长期等待。若题目要求“写者优先”或“公平”，需要增加额外信号量控制读者进入。

同步问题题型提示

许多题目只是经典模型的变形。

常见变形：

题型 对应模型
过独木桥 同方向可共享，反方向互斥，类似读者-写者
展览馆入口/出口 容量限制 + 入口互斥 + 出口互斥
水果盘问题 生产者-消费者 + 不同消费者条件不同
信箱辩论 有界缓冲区 + 双向生产消费

2.6 进程通信

基本概念

并发进程之间的交互必须满足两个基本要求：同步和通信。
互斥是一种特殊的同步，本质上仍然是对进程推进顺序的约束。

进程通信：进程之间的信息交换。

基本操作：

  • 发送操作：send(message)
  • 接收操作：receive(message)

低级通信和高级通信：

类型 特点
低级通信 通信信息量少，通常依赖信号量等机制
高级通信 能交换较多信息，使用共享存储、管道、消息传递等机制

高级通信分类

高级通信方式：

  • 共享存储器系统：通过共享数据结构或共享存储区交换数据
  • 消息传递系统：以格式化消息为单位进行通信
  • 管道通信系统：通过管道以字符流形式传送大量数据

共享存储器系统

共享存储器系统中，多个进程通过共享某些数据结构或共享存储区进行通信。

分类：

  • 基于共享数据结构的通信方式：通信效率低，只适合少量数据
  • 基于共享存储区的通信方式：在内存中划出共享区域，进程通过读写该区域交换大量数据

注意：共享存储只提供“数据放在哪里”，仍需要同步互斥机制保证读写正确。

管道通信系统

管道：用于连接读进程和写进程，以实现它们之间通信的共享文件。
发送进程以字符流形式把大量数据写入管道，接收进程从管道读取数据。

管道特点：

  • 一般是半双工通信
  • 若要双向同时通信，需要建立两条管道
  • 管道本质上是共享文件，需要同步读写
  • 常用于具有亲缘关系的进程间通信

消息传递系统

消息传递系统：进程间通信以格式化消息为单位，程序员通过通信原语实现通信。

直接通信方式

发送进程利用OS提供的发送命令，直接把消息发送给接收进程。

特点：

  • 发送进程和接收进程都显式提供对方标识
  • 通信链路通常自动建立
  • 链路一般只连接一对进程

典型形式：

间接通信方式：信箱通信

发送进程把消息发送到某个中间实体（信箱），接收进程再从信箱中取得消息。

系统常提供的原语：

  • 创建信箱
  • 撤销信箱
  • 发送消息
  • 接收消息

信箱关系：

  • 一对一
  • 多对一
  • 一对多
  • 多对多

消息传递中的若干问题

通信链路：

  • 显式建立链路
  • 隐式建立链路

消息格式：

  • 消息头：发送进程、接收进程、消息长度、类型、编号等控制信息
  • 消息正文：具体数据

通信同步方式：

方式 含义
阻塞发送 发送者发送后等待，直到消息被接收
非阻塞发送 发送者发送后继续执行
阻塞接收 接收者无消息时等待
非阻塞接收 接收者无消息时立即返回

2.7 线程及其实现

线程的基本概念

引入进程是为了使多个程序并发执行，提高资源利用率和系统吞吐量。
引入线程则是为了在保持并发性的同时，减少进程创建、撤销、切换的时空开销。

线程：进程的一部分，描述指令流执行状态。它是进程中的指令执行流的最小单元，是CPU调度的基本单位。

进程的两个基本属性：

  • 进程是拥有资源的独立单位
  • 进程是可独立调度和分派的基本单位

引入线程后，这两个属性被分开：

  • 进程仍是资源拥有单位
  • 线程成为调度和分派单位

线程与进程的比较

对比项 进程 线程
调度 无线程时是调度基本单位 引入线程后是调度和分派基本单位
资源 拥有资源的独立单位 基本不拥有系统资源，只保留必要运行资源
地址空间 不同进程地址空间相互独立 同一进程内线程共享进程地址空间
并发性 进程之间可并发 同一进程内多个线程也可并发
切换开销 较大 较小
独立性 较强 较弱，同进程线程相互影响

线程状态和TCB

线程运行的三个基本状态：

  • 执行状态
  • 就绪状态
  • 阻塞状态

TCB：线程控制块。
类似进程有PCB，系统为每个线程配置TCB，用于保存和管理线程控制信息。

TCB通常包含：

  • 线程标识符
  • 寄存器状态
  • 线程运行状态
  • 优先级
  • 线程专有存储区
  • 栈指针

多线程OS中的进程属性：

  • 进程是资源分配单位
  • 进程包含一个或多个线程
  • 进程不是独立运行单位
  • 同一进程内线程共享进程资源

线程的实现方式

线程实现方式主要有三类：

  • 用户级线程ULT
  • 内核支持线程KST
  • 用户级线程和内核支持线程组合

用户级线程ULT

用户级线程在用户空间中实现。线程创建、撤销、同步与通信等功能不需要内核支持。

优点：

  • 线程切换不需要进入内核空间
  • 调度算法可以由进程专用
  • 实现与操作系统平台关系较弱

缺点：

  • 一个线程执行阻塞系统调用，可能导致整个进程阻塞
  • 内核不知道线程存在，不能把同一进程多个线程调度到多个处理器并行执行

内核支持线程KST

内核支持线程在内核支持下运行，线程创建、撤销、切换由内核完成。

优点：

  • 多处理器系统中，内核可同时调度同一进程中的多个线程并行执行
  • 一个线程阻塞时，内核可调度该进程中的其他线程
  • 内核自身也可采用多线程技术

缺点：

  • 线程切换需要进入内核，开销较大

组合方式

组合方式把用户级线程和内核支持线程结合起来，既保留用户级线程切换快的优点，也允许内核调度多个内核线程。

常见模型：

模型 含义 特点
多对一模型 多个用户线程映射到一个内核线程 管理简单，但一个阻塞可能影响全部
一对一模型 一个用户线程映射到一个内核线程 并行性好，但内核开销较大
多对多模型 多个用户线程映射到多个内核线程 折中方案，兼顾并行和开销
