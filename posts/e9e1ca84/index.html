处理机调度与死锁

本章主线：处理机调度解决 谁先用CPU，死锁处理解决 进程为什么都走不动。前半章看调度层次、准则和算法，后半章看死锁的条件、预防、避免、检测和解除。

3.1 调度的层次

在多道程序环境下，进程数目往往多于处理机数目，它们竞争处理机。系统必须按某种算法，动态地把处理机分配给就绪队列中的某个进程，使之执行。

调度按层次划分：

  • 高级调度：长程调度、作业调度、宏观调度
  • 中级调度：中程调度、交换调度
  • 低级调度：短程调度、CPU调度、进程调度、微观调度

高级调度

高级调度用于决定把外存后备队列中的哪些作业调入内存，并为它们创建进程。

高级调度需要解决：

  • 接纳多少作业进入内存
  • 接纳哪些作业进入内存
  • 使系统保持合适的多道程序度

低级调度

低级调度的主要任务：从就绪队列中选择一个进程来执行，并分配处理机。
它是操作系统中最基本、最频繁的调度。

进程调度功能：

  1. 保存处理机现场信息到PCB中
  2. 按某种算法选择就绪进程占有CPU
  3. 把CPU分配给进程，并恢复其现场

调度方式：

方式 含义 特点
非抢占式 CPU分配给某进程后，直到完成或阻塞才让出 实现简单，响应较差
抢占式 可因时间片、优先级等原因强行剥夺CPU 响应好，开销较大

中级调度

中级调度在内存和外存对换区之间选择进程对换，用于解决内存紧张问题。
它可以把暂时不能运行的进程换出内存，也可以把具备运行条件的进程换入内存。

3.2 调度队列模型和调度准则

调度过程离不开队列。典型队列包括就绪队列、阻塞队列、挂起就绪队列、挂起阻塞队列等。

选择调度方式和算法的准则

面向用户的准则：

  • 周转时间短
  • 响应时间快
  • 截止时间保证
  • 优先权准则

面向系统的准则：

  • 系统吞吐量高
  • 处理机利用率高
  • 各类资源平衡利用
  • 公平性和可预测性

常用指标：

指标 公式/含义
CPU利用率 CPU有效工作时间 / CPU总运行时间
周转时间 作业完成时间 - 作业提交时间
带权周转时间 周转时间 / 实际服务时间
等待时间 作业或进程在队列中等待的时间
响应时间 从提交请求到首次响应的时间

3.3 调度算法

先来先服务FCFS

基本思想：按照进程进入就绪队列的先后次序分配处理机。
一般采用非抢占式调度。

优点：

  • 简单
  • 对长作业有利
  • 实现开销小

缺点：

  • 对短作业不利
  • 可能使平均等待时间较长
  • 不适合分时系统和实时系统

短作业/短进程优先SJF/SPF

基本思想：优先选择估计运行时间最短的作业或进程。

类型：

  • 非抢占式SJF/SPF：一旦运行，直到完成或阻塞
  • 抢占式SJF/SPF：新到达进程若剩余时间更短，可抢占CPU

优点：能有效降低平均等待时间和平均周转时间。
缺点：对长作业不利，可能产生饥饿；要求预知运行时间。

时间片轮转RR

应用于分时系统，保证用户请求能得到及时响应。
系统把CPU时间划分为固定长度的时间片，就绪队列中的进程轮流获得时间片执行。

时间片选择需要考虑：

  • 系统对响应时间的要求
  • 就绪队列中进程数目
  • 系统处理能力
  • 进程切换开销

时间片过大：退化为FCFS。
时间片过小：进程切换频繁，系统开销大。

优先权调度算法

系统根据进程优先级选择进程。

类型：

  • 非抢占式优先权算法
  • 抢占式优先权算法
  • 静态优先权
  • 动态优先权

静态优先权：创建进程时确定，运行期间不变。
动态优先权：运行过程中可根据等待时间、资源需求等因素调整。

高响应比优先HRRN

FCFS只考虑等待时间，SJF只考虑服务时间，高响应比优先算法兼顾二者。

响应比：

特点：

  • 短作业容易获得较高响应比
  • 长作业等待时间增加后也会逐渐提高优先级
  • 不会长期饥饿

多级反馈队列调度算法

多级反馈队列是时间片轮转和优先级调度的综合。

基本思想：

  • 设置多个就绪队列，优先级由高到低
  • 高优先级队列时间片较短，低优先级队列时间片较长
  • 新进程先进入最高优先级队列
  • 一个时间片内未完成，则降到下一队列
  • 高优先级队列为空时，才调度低优先级队列

特点：

  • 长、短作业兼顾
  • 有较好的响应时间
  • 不必事先估计执行时间

3.4 死锁的基本概念

死锁：多个进程在运行过程中因争夺资源而造成的一种僵局。若无外力作用，这些进程都无法向前推进。

产生死锁的原因

资源分类：

  • 可重用资源：处理机、内存、I/O设备、文件等
  • 消耗性资源：消息、信号、中断等

死锁产生原因：

  • 竞争不可剥夺资源
  • 竞争临时性资源
  • 进程推进顺序不当

产生死锁的必要条件

死锁必须同时具备四个Coffman条件：

条件 含义
互斥条件 某资源一段时间内只能被一个进程占用
请求和保持条件 进程已保持至少一个资源，又请求新的资源
不可剥夺条件 已获得资源在未使用完前不能被强行剥夺
环路等待条件 存在进程资源循环等待链

处理死锁的基本方法

  • 预防死锁：破坏产生死锁的必要条件
  • 避免死锁：动态检查资源分配后系统是否安全
  • 检测死锁：允许死锁发生，然后检测出来
  • 解除死锁：检测到死锁后采取措施恢复

3.5 死锁的预防和避免

死锁预防

死锁预防通过破坏四个必要条件之一来避免死锁。

破坏条件 方法 代价
互斥条件 将独占资源改为共享资源 很多资源无法共享
请求和保持条件 一次性申请全部资源 资源利用率低，可能饥饿
不可剥夺条件 申请不到新资源时释放已占资源 实现复杂，代价较高
环路等待条件 按资源编号顺序申请 编号和使用顺序受限制

死锁避免

死锁避免允许进程动态申请资源，但系统在分配前先计算安全性。只有分配后仍处于安全状态，才真正分配资源。

安全状态：系统能找到某种进程序列，使每个进程都能获得所需资源并顺利完成。
安全序列：能使全部进程顺利完成的进程执行顺序。

基本事实：

  • 系统处于安全状态，一定不会死锁
  • 系统处于不安全状态，不一定死锁，但可能死锁
  • 死锁状态一定是不安全状态

银行家算法

银行家算法是典型的死锁避免算法。

数据结构：

名称 含义
Available 系统当前可用资源向量
Max 每个进程对各类资源的最大需求矩阵
Allocation 每个进程已分配资源矩阵
Need 每个进程尚需资源矩阵，Need = Max - Allocation
Request 某进程本次请求资源向量

资源分配算法：

  1. 若Request大于Need，说明请求超过最大需求，出错
  2. 若Request大于Available，说明资源不足，等待
  3. 试探性分配资源
  4. 调用安全性算法
  5. 若安全则正式分配，否则恢复原状态并等待

安全性检查算法：

  1. Work = Available
  2. Finish[i] = false
  3. 找到满足Finish[i] = false且Need[i] <= Work的进程
  4. 令Work = Work + Allocation[i]，Finish[i] = true
  5. 若所有Finish都为true，则系统安全

3.6 死锁的检测和解除

如果系统既不采用死锁预防，也不采用死锁避免，而是直接分配资源，则必须提供死锁检测和解除机制。

资源分配图

资源分配图又称进程-资源图，用于描述进程和资源之间的请求与分配关系。

重要结论：

  • 若资源分配图中不存在环路，则系统无死锁
  • 若资源分配图中存在环路，且每类资源只有一个实例，则有死锁
  • 若每类资源有多个实例，有环路不一定死锁

资源分配图化简

如果经过一系列化简，能消去图中所有边，使所有进程都成为孤立结点，则该图可完全简化。
若不可完全简化，则系统处于死锁状态。

死锁定理：S为死锁状态的充分条件是，当且仅当S状态的资源分配图不可完全简化。

死锁检测算法

死锁检测算法类似银行家算法中的安全性检查。

基本思想：
获得某时刻系统中各类可利用资源向量，对进程集合进行模拟推进。若某些进程最终无法完成，则这些进程可能处于死锁。

死锁解除

常见方法：

  • 剥夺资源：从某些死锁进程中剥夺资源，分配给其他进程
  • 撤销进程：撤销一个或多个死锁进程

选择被撤销或被剥夺资源的进程时，应考虑：

  • 进程优先级
  • 已执行时间和剩余时间
  • 已占资源数量
  • 进程类型：交互型、批处理型等
  • 解除死锁的代价
